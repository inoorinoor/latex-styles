Системы управления идентификацией и доступом (Identity and Access Management, IAM) 
являются критически важным компонентом современных распределённых приложений. 
Среди таких систем особое место занимает Keycloak --- решение с открытым исходным кодом, 
предоставляющее механизмы аутентификации, авторизации, управления пользователями и единого 
входа (SSO). Keycloak широко применяется в корпоративных системах, микросервисных архитектурах
и облачных платформах.


По умолчанию Keycloak поддерживает реляционные СУБД: PostgreSQL, MySQL, MariaDB, Oracle,
TiDB и Microsoft SQL Server. Хранение данных в реляционных СУБД в условиях высоких нагрузок
и масштабирования может становиться узким местом системы: традиционные РСУБД ограничены 
в горизонтальном масштабировании и требуют сложных решений для обеспечения отказоустойчивости.

YDB (Yandex Database) --- это распределённая СУБД, разработанная компанией Яндекс и 
выпущенная как программное обеспечение с открытым исходным кодом. YDB обеспечивает 
горизонтальное масштабирование, высокую доступность и строгую транзакционную семантику
при распределённом выполнении. Архитектура YDB изначально ориентирована на работу в
условиях высоких нагрузок, что делает её перспективной заменой традиционным РСУБД в
сценариях с требованиями к масштабируемости.

Однако YDB имеет ряд архитектурных особенностей, отличающих её от стандартных реляционных 
СУБД: 
\begin{itemize}
    \item отсутствие поддержки внешних ключей на уровне базы данных;
    \item обязательное наличие явного первичного ключа в каждой таблице;
    \item специфический SQL-диалект YQL;
    \item отсутствие поддержки пессимистичных блокировок (`FOR UPDATE`);
    \item YDB использует optimistic concurrency control для обнаружения конфликтов с другими выполняющимися транзакциями;
\end{itemize}

Целью работы является разработка программного модуля, обеспечивающего интеграцию Keycloak
с YDB в качестве основного хранилища данных с сохранением функциональной совместимости
и корректности работы системы.

Практическая значимость работы состоит в том, что разработанное расширение позволяет
использовать распределённую СУБД YDB в качестве хранилища данных для Keycloak, 
что открывает возможности для горизонтального масштабирования IAM-компонента в крупных
системах.

\section{Система управления идентификацией Keycloak}

Keycloak --- это система управления идентификацией и доступом с открытым исходным кодом, разработанная компанией Red Hat.
Система реализует протоколы OAuth 2.0, OpenID Connect и SAML 2.0, и предоставляет единую точку входа для множества приложений.

\subsection{Архитектура Keycloak}

Keycloak построен на основе Quarkus и использует Hibernate в качестве ORM-фреймворка для работы с базой данных.
Управление схемой базы данных осуществляется через Liquibase.

Центральным элементом расширяемости Keycloak является механизм \textbf{Service Provider Interface (SPI)}.
SPI позволяет заменять внутренние реализации компонентов пользовательскими, сохраняя совместимость с остальными частями системы.

\subsubsection{Выбор провайдера по приоритету}

Алгоритм выбора следующий:

\begin{enumerate}
    \item Если явно задан провайдер через конфигурацию, то используется он; 
    \item Если зарегистрирована только одна фабрика, то используется она;
    \item Если зарегистрировано несколько фабрик, то выбирается та, у которой наибольшее значение `order()` среди всех с `order() > 0`. 
    \item Иначе используется фабрика с идентификатором default. 
\end{enumerate}


\subsection{Ключевые SPI и ProviderFactory для работы с базой данных}

\begin{itemize}
    \item `JpaConnectionProviderFactory` --- управляет созданием соединений с базой данных и жизненным циклом JPA `EntityManagerFactory`;
    \item `LiquibaseConnectionProviderFactory` --- управляет подключением Liquibase для выполнения миграций;
    \item `DBLockProviderFactory` --- реализует распределённую блокировку для синхронизации при параллельном старте нескольких инстанций Keycloak;
    \item `RealmProviderFactory`, `ClientProviderFactory`, `ClientScopeProviderFactory` и тд --- фабрики провайдеров для работы с основными сущностями данных.
\end{itemize}

\subsection{Модель данных Keycloak}

Keycloak хранит данные в реляционных таблицах, отображаемых через Hibernate-аннотации на JPA-сущности.

Основные сущности включают:

\begin{itemize}
    \item \textbf{Realm} --- область, содержащая конфигурацию, пользователей, клиентов, роли и группы; 
    \item \textbf{Client} --- OAuth/OIDC-клиент, зарегистрированный в realm; 
    \item \textbf{ClientScope} --- область видимости, определяющая набор claims в токене; 
    \item \textbf{Role} --- роль, назначаемая пользователям или клиентам; 
    \item \textbf{Group} --- группа пользователей с наследуемыми ролями; 
    \item \textbf{UserEntity} --- пользователь с атрибутами, учётными данными и привязкой к внешним провайдерам идентификации;
\end{itemize}

Cхема Keycloak включает более 80 таблиц и активно использует внешние ключи для обеспечения ссылочной целостности. 
Это является принципиальным ограничением при переносе на YDB.

\section{СУБД YDB}
YDB --- это NewSQL-база данных, разработанная Яндексом и опубликованная как открытое программное обеспечение в 2022 году. 
YDB сочетает свойства реляционных СУБД (строгая типизация, SQL-запросы, ACID-транзакции) с возможностями распределённых 
систем (горизонтальное масштабирование, автоматическое шардирование).

\subsubsection{Ключевые свойства YDB}

\textbf{Транзакционная модель.} YDB поддерживает сериализуемую изоляцию транзакций.
Однако при конкурентном изменении одних и тех же строк сервер может вернуть ошибку `Transaction lock invalidated`,
сигнализирующую о конфликте транзакций. Такие ошибки относятся к классу *retryable* --- приложение должно повторить
транзакцию с нуля.

\textbf{Отсутствие поддержки FOR UPDATE.} YDB не реализует пессимистичные блокировки строк.
Запросы с `SELECT ... FOR UPDATE` или `SELECT ... FOR SHARE` являются синтаксически некорректными в YQL.

\textbf{Обязательный первичный ключ.} Каждая таблица в YDB обязана иметь явно определённый первичный ключ.
Таблицы без первичного ключа создать невозможно.

\textbf{Отсутствие внешних ключей.} YDB не поддерживает декларативные ограничения внешних ключей.
Ссылочная целостность должна обеспечиваться на уровне приложения.

\textbf{Типы данных.} Основные строковые типы в YDB --- `Utf8` (строки в кодировке UTF-8) и `Bytes` (бинарные данные).
В отличие от SQL Server/PostgreSQL, тип `String` в YQL является синонимом `Bytes`, а не текстового типа.

\textbf{Диалект YQL.} YDB использует YQL (Yandex Query Language) --- диалект SQL с рядом ограничений:

\begin{itemize}
    \item не поддерживаются ссылки на столбцы по порядковому номеру (`ORDER BY 1`);
    \item не поддерживается `CONCAT()` --- используется оператор `||`;

\end{itemize}
